## 二、发现的问题与优化建议

### 2.1 🔴 严重问题

#### 1. PowerShell 脚本注入风险 (L1254-1296)
```rust
format!("... --busid {busid} ...")
```
**问题**: `busid` 直接插入 PowerShell 脚本字符串，虽然 `busid` 来自 usbipd list 解析（相对可信），但缺少转义/验证。

**建议**: 添加 busid 格式验证（正则 `^\d+-\d+$`），或使用参数化调用。

#### 2. Mutex 中毒处理不一致 (多处)
```rust
state.ports.lock().unwrap_or_else(|e| e.into_inner())
```
**问题**: 部分地方用 `unwrap_or_else` 恢复中毒锁，部分直接 `unwrap()`（如 L1005, L1041-1044）。中毒锁说明前一个持有者 panic 了，简单恢复可能导致数据不一致。

**建议**: 统一策略 - 要么全部 `unwrap_or_else`，要么记录日志后恢复。

#### 3. `open_wsl_serial` 中的竞态窗口 (L1005)
```rust
{ let mut s = state.sessions.lock().unwrap(); if let Some(old) = s.remove(&monitor_id) { ... } }
// ← 此处释放锁后，另一个线程可能插入同 key 的 session
let distro = get_or_start_wsl_distro()?;
```
**问题**: 两段操作不在同一临界区内，理论上可能重复打开。

**建议**: 整个 `open_wsl_serial` 逻辑用单一 Mutex 保护，或使用 `entry` API。

### 2.2 🟡 性能优化

#### 4. `get_wsl_distributions` N+1 子进程问题 (L806-830)
```rust
for distro in distros {
    // 每个发行版 spawn 2 个子进程: cat /proc/uptime + free -m
}
```
**问题**: 3 个发行版 = 6 个子进程，每次调用耗时可能超过 1 秒。

**建议**:
- 合并命令: `wsl -d X -- sh -c "cat /proc/uptime; free -m"`
- 或使用 `wsl_shell_exec` 复用持久化 shell

#### 5. `check_wsl_running` 每次 spawn 新线程 (L655-698)
```rust
std::thread::spawn(move || { ... });
rx.recv_timeout(Duration::from_secs(5))
```
**问题**: 线程创建开销 + 5 秒超时可能阻塞调用者。

**建议**: 改用 `std::process::Command::output()` 直接同步调用（WCL 已在后台），或用全局缓存+TTL 减少调用频率。

#### 6. `start_wsl_watcher` 2 秒轮询 (L714-741)
**问题**: 每 2 秒调用 `check_wsl_running()`，持续消耗 CPU。

**建议**:
- 增加轮询间隔到 5-10 秒（WSL 状态变化不频繁）
- 或改用事件驱动（监听 WSL 进程退出事件）

#### 7. `list_wsl_devices` 重复解析 usbipd list (L487-594)
**问题**: `list_wsl_devices` 和 `attach_port_to_wsl_blocking` 都解析 usbipd list 输出，逻辑重复。

**建议**: 提取共享的 `parse_usbipd_list()` 函数。

### 2.3 🟢 代码质量

#### 8. 错误消息不一致
- 中文: `"打开失败: {}"`, `"未连接串口"`, `"关闭 WSL 发行版失败: {}"`
- 英文: `"bridge 进程已退出"`, `"port not open"`
- 混合: `"读取 bridge 响应失败: {}"`

**建议**: 统一为中文（面向国内用户），或提供错误码。

#### 9. 未使用的 import 和变量
- L256: `product_name` 声明后未赋值（始终为空字符串）
- L569: `has_com` 变量在 L570 重新计算

**建议**: 清理冗余代码。

#### 10. `deploy_bridge` 可能的竞态 (L940-961)
```rust
if let Ok(o) = check { if o.status.success() { return Ok(()); } }
// ← 检查和写入之间，其他进程可能修改文件
```
**建议**: 使用原子写入（先写临时文件再 rename）。

## 三、逻辑合理性评估

### ✅ 合理的设计

1. **串口状态管理**: 用 `HashMap<String, Box<dyn SerialPort>>` + Mutex 管理多串口，清晰且线程安全。

2. **WSL Bridge 架构**: stdin/stdout JSON 管道通信，避免网络依赖，进程生命周期可控。

3. **设备插拔检测**: `CM_Register_Notification` 是 Windows 推荐方式，比轮询高效。

4. **异步 WSL 操作**: `shutdown_wsl` 改用 `spawn_blocking` 避免阻塞 UI，正确。

5. **配置持久化**: `%APPDATA%\seahi-serial\config.json` 符合 Windows 规范。

### ⚠️ 需要关注

1. **WSL Shell 锁持有时间** (L131-170): `wsl_shell_exec` 在读取循环期间持有 reader 锁，可能阻塞其他命令。

2. **`attach_port_to_wsl_blocking` 复杂度** (L1130-1342): 200+ 行函数，嵌套 5 层 if/else，可读性差。

3. **`is_process_alive` 硬编码 exit code** (L709): `code == 259` (STILL_ACTIVE)，但如果进程以其他非零码退出也会返回 false。

4. **`open_wsl_serial` 的 ready 检测** (L1014-1021): 阻塞等待 stderr 输出 "ready"，无超时机制，可能永久阻塞。
