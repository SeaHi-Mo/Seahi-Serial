# 串口自动化工作流功能实现方案

## 目标

为 Seahi Serial 添加"自动化工作流"功能：用户可为每个监视器配置自定义规则，当收到匹配的串口数据时自动执行响应动作。类似钉钉自动化的 if-this-then-that 模式。

## 架构决策

**核心思路：前端轮询驱动 + Rust 后端匹配执行**

当前架构是前端 100ms 轮询 `read_data`。不引入后台读取线程（`serialport` crate 不支持 `Box<dyn SerialPort>` 的并发读取拆分），而是在现有轮询循环中增加一次 `invoke('check_workflow_matches')` 调用。匹配逻辑在 Rust 中执行（使用 `regex` crate），动作也在 Rust 中异步执行。

数据流：
```
前端轮询 read_data (100ms)
  → 收到数据 → 显示到 UI
  → 调用 check_workflow_matches(data)
      → Rust: 遍历规则, 匹配条件
      → 匹配成功 → 在独立线程中执行动作序列(含延时)
      → 返回匹配到的规则名列表 → 前端显示 [Workflow] 触发提示
```

## 一、数据模型

### 配置 JSON（config.json v2）

```jsonc
{
  "version": 2,                    // 从 1 升级
  "extraCount": 0,
  "monitors": {
    "main": {
      // ...现有字段不变...
      "workflows": [               // 新增：per-monitor 工作流规则
        {
          "id": "wf_1720000000000",
          "name": "AT命令自动应答",
          "enabled": true,
          "conditions": [
            { "type": "string_contains", "value": "AT+CMD" }
          ],
          "actions": [
            { "type": "send_data", "data": "OK\r\n", "encoding": "text", "delayBefore": 0 },
            { "type": "toggle_dtr_rts", "signal": "dtr", "level": true, "delayBefore": 50 }
          ]
        }
      ]
    }
  }
}
```

### 条件类型

| type | 匹配方式 | value 格式 |
|------|----------|-----------|
| `string_contains` | UTF-8 丢弃解码后子串匹配 | `"OK"`, `"AT+RST"` |
| `regex` | UTF-8 丢弃解码后正则匹配 | `"\\+CME ERROR: \\d+"` |
| `exact_bytes` | 原始字节窗口匹配（滑动窗口） | `"FF 01 02"` 或 `"FF0102"` |

条件之间 **AND** 关联：全部匹配才触发。

### 动作类型

| type | 说明 | 参数 |
|------|------|------|
| `send_data` | 发送数据 | `data`, `encoding` ("text"/"hex"), `delayBefore` (ms) |
| `toggle_dtr_rts` | 切换信号线 | `signal` ("dtr"/"rts"), `level` (bool), `delayBefore` (ms) |
| `save_log` | 追加保存日志 | `delayBefore` (ms)，保存到监视器的日志目录 |

动作按数组顺序执行，`delayBefore` 表示执行该动作前等待的毫秒数。动作在独立线程中执行，不阻塞前端轮询。

struct WorkflowCondition {

    encoding: String,
    #[serde(default)]
    signal: String,
    #[serde(default)]
    level: bool,
    #[serde(default)]
    delay_before: u64,
}

#[derive(Debug, Clone, serde::Deserialize, serde::Serialize)]
struct WorkflowRule {
    id: String,
    name: String,
    enabled: bool,
    conditions: Vec<WorkflowCondition>,
    actions: Vec<WorkflowAction>,
}

struct WorkflowState {
    rules: Mutex<HashMap<String, Vec<WorkflowRule>>>,
    log_dirs: Mutex<HashMap<String, String>>,  // monitor_id -> log_dir
}

### 2.3 匹配引擎函数

```rust
fn match_condition(cond: &WorkflowCondition, raw: &[u8]) -> bool {
    match cond.cond_type.as_str() {
        "string_contains" => {
            String::from_utf8_lossy(raw).contains(&cond.value)
        }
        "regex" => {
            let text = String::from_utf8_lossy(raw);
            Regex::new(&cond.value).map(|re| re.is_match(&text)).unwrap_or(false)
        }
        "exact_bytes" => {
            let expected = parse_hex_bytes(&cond.value);
            if expected.is_empty() { return false; }
            raw.windows(expected.len()).any(|w| w == expected.as_slice())
        }
        _ => false,
    }
}

fn parse_hex_bytes(s: &str) -> Vec<u8> {
    let cleaned: String = s.chars().filter(|c| !c.is_whitespace()).collect();
    cleaned.as_bytes().chunks(2)
        .filter_map(|chunk| u8::from_str_radix(std::str::from_utf8(chunk).ok()?, 16).ok())
        .collect()
}
```

### 2.4 动作执行（在独立线程中调用）

```rust
fn execute_actions(
    actions: &[WorkflowAction],
    monitor_id: &str,
    port_state: &PortState,
    wsl_state: &WslSerialState,
    port_type: &str,
    log_dir: &str,
    received: &[u8],
) {
    for action in actions {
        if action.delay_before > 0 {
            std::thread::sleep(Duration::from_millis(action.delay_before));
        }
        match action.action_type.as_str() {
            "send_data" => {
                let bytes = if action.encoding == "hex" {
                    parse_hex_bytes(&action.data)
                } else {
                    action.data.as_bytes().to_vec()
                };
                match port_type {
                    "wsl" => { /* 通过 bridge_command 发送 */ }
                    _ => {
                        if let Some(port) = port_state.ports.lock().unwrap_or_else(|e| e.into_inner()).get_mut(monitor_id) {
                            let _ = port.write_all(&bytes);
                        }
                    }
                }
            }
            "toggle_dtr_rts" => { /* set_dtr / set_rts 逻辑 */ }
            "save_log" => { /* 追加写入 log_dir/workflow_log.txt */ }
            _ => {}
        }
    }
}
```

### 2.5 新增 Tauri 命令

| 命令 | 参数 | 返回 | 用途 |
|------|------|------|------|
| `check_workflow_matches` | `monitor_id, data: Vec<u8>, port_type` | `Vec<Value>` (匹配的规则名) | 前端轮询时调用，匹配+执行 |
| `save_workflows` | `monitor_id, workflows_json` | `Result` | 保存工作流规则到内存 |
| `load_workflows` | `monitor_id` | `String` (JSON) | 加载工作流规则 |
| `init_workflows` | `config_json` | `Result` | 启动时从配置初始化 |
| `update_workflow_log_dir` | `monitor_id, log_dir` | `Result` | 更新日志目录 |

`check_workflow_matches` 核心逻辑：
1. 从 `WorkflowState.rules` 读取该 monitor 的规则
2. 遍历 enabled 的规则，AND 匹配所有条件
3. 匹配成功 → `std::thread::spawn` 执行动作（不阻塞当前调用）
4. 返回匹配到的规则列表供前端显示提示

## 六、实施顺序

1. **Rust 数据结构 + 匹配引擎** — 结构体、`match_condition`、`parse_hex_bytes`
2. **Rust 动作执行** — `execute_actions`（含线程池/spawn）
3. **Rust Tauri 命令** — `check_workflow_matches`、`save_workflows`、`load_workflows`、`init_workflows`、`update_workflow_log_dir`
4. **注册到 Builder** — `WorkflowState` + 命令注册
5. **前端配置迁移** — `loadAndApplyConfig` 版本检查 + V1→V2
6. **前端数据流集成** — `startReading`/`startWslReading` 中调用 `checkWorkflowMatches`；`collectConfig`/`applyMonitorConfig` 加入 workflows
7. **前端 UI** — CSS 样式 + 工作流面板 HTML + 规则渲染
8. **前端 JS** — 工作流 CRUD 函数 + 测试功能

## 七、验证方式

1. `npm run dev` 启动开发模式
2. 打开一个串口连接，配置一条工作流规则（如：收到 "hello" 自动回复 "world"）
3. 用另一个串口工具发送 "hello"，验证自动回复
4. 测试正则匹配和 HEX 精确匹配
5. 测试延时动作、DTR/RTS 切换、日志保存
6. 重启应用，验证工作流配置持久化
7. `npm run build` 验证 Release 构建通过
