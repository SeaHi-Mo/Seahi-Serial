# Seahi-Serial 专家启发式走查报告

> **研究对象**：Seahi-Serial v0.2.0（Tauri 2 + Rust 串口调试器）  
> **研究方法**：真实启动软件 + 界面截图 + 前端源码分析 + 启发式评估  
> **研究日期**：2026-07-18  
> **研究者**：UX Researcher  
> **文档版本**：v1.0

---

## 一、重要免责声明

本报告**不是用户研究结论**，而是**基于真实界面的专家启发式走查**（Heuristic Evaluation）。所有发现必须由真实用户研究进一步验证。

**证据来源**:
- **E1 真实界面截图**：捕获了 release 与 dev 两种构建的启动界面（见 `.walkthrough/shots/`）。
- **E2 前端源码**：完整阅读 `src/index.html`（约 4400 行），了解界面结构、控件层级、交互触发逻辑。
- **E3 测试用例**：对照 `TEST_CASES.md` 评估界面操作路径。
- **E4 项目文档**：`README.md`、`RELEASE_NOTES.md`、`HANDOVER.md` 与代码评审记录。

**环境限制**：本机会话中无法向 Tauri WebView 成功注入鼠标/键盘事件（Python/pyautogui 的输入事件未生效），因此未进行动态交互测试（如下拉展开、表单输入、面板切换的实际操作）。所有关于"动态交互"的推断均来自源码阅读，而非实际操作观察。

## 二、执行摘要（关键发现）

| 编号 | 发现 | 证据强度 | 对原假设的影响 |
|------|------|---------|--------------|
| **F1** | 高级设置行（数据位/停止位/校验位/DTR/RTS）**默认全部展开**，信息密度极高。对专家用户友好，对新手可能是信息过载。 | 强（E1+E2） | 修正 P-01：不是"隐藏较深"，而是"全部摊开导致主次不分" |
| **F2** | **快捷指令入口**仅显示为一个三横线图标（无文字标签），位于发送栏最右侧，可发现性明显不足。 | 强（E1+E2） | 支撑原假设 P-01（高级功能可发现性低） |
| **F3** | **自动化工作流**入口在高级设置行最右侧，以 tiny 文字"＋ 添加规则"呈现，没有功能说明。 | 强（E1+E2） | 支撑 H3（工作流真实使用率可能低于 20%） |
| **F4** | **WSL 端口映射**在全局标题栏有文字按钮，但视觉权重低，未体现其作为差异化核心功能的地位。 | 中（E1+E2） | 支撑 P-02（WSL 门槛高，且入口不够突出） |
| **F5** | 主输出区域为**完全空白**，没有任何空状态提示或引导，首次用户难以预判此处会显示什么。 | 强（E1） | 新增痛点：空状态缺失 |
| **F6** | 主题风格切换（6 种艺术风格）在全局标题栏非常醒目，与核心串口功能并列，可能喧宾夺主。 | 强（E1） | 支撑 H6（主题风格使用率可能极低） |
| **F7** | 端口下拉已正确显示中文友好名（如 "WCH-Link SERIAL (COM29)" / "USB-Enhanced-SERIAL CH343 (COM48)"），这是显著优势。 | 强（E1） | 删除原假设中"中文乱码"风险 |
| **F8** | 工具栏在 1367px 宽度下仍采用两排布局，窄屏或分栏后可能严重拥挤。 | 中（E1） | 新增布局适应性风险 |
| **F9** | 没有明显的键盘快捷键提示、没有 ARIA 标签、大量图标依赖悬停 tooltip，辅助功能薄弱。 | 中（E2） | 新增辅助功能/包容性设计风险 |
| **F10** | "更多设置"齿轮按钮在图标组最末，但高级设置行已经展开——按钮当前状态与系统状态不一致，可能造成困惑。 | 中（E1+E2） | 新增状态一致性风险 |

## 三、走查方法与环境

### 3.1 执行步骤

1. 启动 `src-tauri/target/release/seahi-serial.exe` 与 `npm run dev` 两种构建。
2. 使用 Python + ctypes + Windows GDI 捕获真实窗口 PNG 截图。
3. 尝试用 Python/pyautogui 进行动态交互（点击、输入、F12），但未成功向 WebView 注入输入事件。
4. 阅读 `src/index.html` 源码，补充交互流程、状态变化、DOM 结构信息。
5. 对照 `TEST_CASES.md` 的 10 类用例，逐项评估界面可用性。

### 3.2 环境限制的影响

| 能力 | 是否实现 | 局限 |
|------|---------|------|
| 启动真实软件 | ✅ | 两个构建均正常启动 |
| 捕获真实界面 | ✅ | 获得 1367×1035 清晰窗口截图 |
| 动态点击/输入 | ❌ | Python/pyautogui 输入事件未生效 |
| 打开 devtools | ❌ | F12 未生效，无法执行 JS 触发 onboarding/状态变化 |
| 源码静态分析 | ✅ | 可推断各功能的界面结构与交互逻辑 |

因此，本走查在**界面结构、信息架构、可发现性、视觉层级**上有较高置信度，在**动态反馈、错误提示、动画过渡**上置信度较低。

## 四、按功能域的详细发现

### 4.1 基础串口配置（TC-1.1 ~ TC-1.5）

**观察**:
- 端口下拉已显示友好名（如 `WCH-Link SERIAL (COM29)`），中文解析正确，这是明确优势（F7）。
- 波特率输入框与下拉箭头并存，支持直接输入和快速选择。
- 高级设置行默认展开：数据位、停止位、校验位、DTR、RTS 全部可见。

**问题**:
- **F1 信息密度过高**：对 80% 使用默认配置的用户来说，DTR/RTS/校验位等参数占用了宝贵的首屏空间。
- **F10 状态不一致**：齿轮按钮在高级设置行已经展开的情况下仍显示"展开"图标，用户可能不知道点击它会折叠。
- **缺少"恢复默认"或"常用配置模板"**：如果用户切换项目，需要手动重置多个参数。

**建议**:
- 高级设置行默认折叠，仅保留最常用参数（波特率、行尾、查看模式、开始按钮）。
- 增加"常用配置"快速切换（如 115200/8N1、9600/8N1）。
- 齿轮按钮的状态应与高级设置行的展开状态实时同步。

### 4.2 数据收发（TC-1.6 ~ TC-1.9）

**观察**:
- 查看模式切换（文本/HEX）在工具栏左侧，相对明显。
- 发送模式下拉（文本/HEX）在输入栏右侧，有文字标签，可发现性尚可。
- 自动滚动、时间戳、行号、回显、终端模式以图标按钮呈现，依赖 tooltip。

**问题**:
- **F5 空状态缺失**：大输出区域没有任何提示，首次用户不知道"这里会显示串口数据"。
- **图标可发现性不足**：终端模式、消息回显、行号等功能的图标含义不直观，新手需要逐个悬停才能理解。
- **HEX 输入格式缺少引导**：输入框 placeholder 仅提示"输入要发送的内容"，未说明 HEX 格式要求。

**建议**:
- 在空输出区添加提示文案："连接串口后，收到的数据将显示在这里。"
- 在输入框 placeholder 中根据当前发送模式动态提示（如 HEX 模式显示"例：48 65 6C 6C 6F"）。
- 对不常用图标按钮，增加文字标签或图标+文字组合。

### 4.3 快捷指令（TC-2.1 ~ TC-2.2）

**观察**:
- 快捷指令入口在发送栏最右侧，是一个三横线图标按钮，无文字标签。
- 源码显示默认有 5 条空白快捷指令（"指令1"~"指令5"）。

**问题**:
- **F2 可发现性严重不足**：用户可能完全不知道这个按钮是快捷指令入口。
- 默认空白指令会强化"这个功能没准备好"的印象。

**建议**:
- 将按钮改为"快捷指令 ▼"文字+图标组合。
- 预置 1-2 条示例指令（如 `"AT\r\n"`），降低首次使用门槛。
- 在引导中专门展示快捷指令功能。

### 4.4 日志功能（TC-3.1 ~ TC-3.2）

**观察**:
- 日志目录选择、保存日志、复制全部功能集中在高级设置行。
- 清空输出按钮在工具栏图标组中（扫帚图标）。

**问题**:
- 日志保存入口被折叠在高级设置行中，对于量产测试场景不够快捷。
- 缺少日志命名规则、自动按日期/设备归档的界面提示。

**建议**:
- 将"保存日志"提升为发送栏附近常驻按钮（尤其量产测试用户高频使用）。
- 增加日志自动命名规则设置（如 `{日期}_{端口}.txt`）。

### 4.5 多监视器（TC-4.1）

**观察**:
- "打开额外监视器"在全局标题栏左侧，有明确文字。
- 源码中每个监视器是独立 DOM 结构，配置独立。

**问题**:
- **F8 分栏后的布局拥挤**：当前窗口宽度 1367px 下工具栏已占两排，分栏后每个监视器宽度减半，高级设置行可能无法完整显示。
- 多监视器之间没有明显的视觉区分（如颜色标签、编号）。

**建议**:
- 分栏时自动折叠高级设置，或提供"仅显示必要控件"的紧凑模式。
- 为每个监视器增加可编辑标题或颜色标识。

### 4.6 WSL 端口映射（TC-5.1 ~ TC-5.4）

**观察**:
- WSL 入口在全局标题栏，文字按钮"WSL 端口映射"。
- 点击后切换到独立面板，主监视器隐藏。

**问题**:
- **F4 入口不够突出**：WSL 是该工具的重要差异化功能，但按钮视觉权重与"提交issue"相当。
- 依赖 usbipd-win、管理员权限、Python bridge，门槛本身已经很高；如果入口再不明显，用户更难走到这一步。
- 在 v0.2.0 的默认首屏中，WSL 与自动化工作流等高级功能混在一起。

**建议**:
- 为 WSL 功能设计独立入口或状态指示器（如检测到 WSL 环境时点亮入口）。
- 在 WSL 面板首次打开时提供分步检查清单（usbipd 是否安装？WSL 是否运行？）。
- 在引导中明确演示 WSL 场景。

### 4.7 主题系统（TC-6.1 ~ TC-6.2）

**观察**:
- 右上角有主题切换开关和主题风格下拉（6 种风格）。
- 风格下拉在全局标题栏非常醒目，位于"提交issue"与主题开关之间。

**问题**:
- **F6 喧宾夺主**：6 种艺术风格（浮世绘彩、诗意东方、水墨丹青等）对开发者工具而言可能是过度设计。
- 主题切换和风格切换两个入口在相邻位置，用户可能混淆"主题"与"风格"的差异。

**建议**:
- 问卷中重点验证：用户实际切换风格的频率与偏好。
- 考虑将风格选择降级到设置菜单，仅在标题栏保留深色/浅色切换。

### 4.8 窗口行为与分栏布局（TC-7.1 ~ TC-7.2）

**观察**:
- 自定义标题栏，集成窗口控制按钮。
- 分栏时通过 pane-container 添加 monitor-pane。

**问题**:
- **F8 布局拥挤**：工具栏已占两排，分栏后空间不足。
- 没有明显的"当前是第几个监视器"的视觉标识。

### 4.9 配置持久化与首次引导（TC-8.1 ~ TC-8.2）

**观察**:
- 源码中有 9 步 onboarding 结构（`onboarding-overlay` + `onboard-step`）。
- 实际启动时未触发 onboarding（localStorage 已标记完成）。

**问题**:
- 无法通过截图验证 onboarding 的实际聚光灯效果与步骤长度。
- 9 步引导可能过长，用户容易在前几步跳过。
- 无法在本会话中清除 localStorage 重新触发。

**建议**:
- 在真实用户测试中重点观察 onboarding 的完成率与跳过步数。
- 考虑将 9 步压缩为 4-5 步，或改为"使用时按需提示"的渐进式引导。

### 4.10 异常场景（TC-10.1 ~ TC-10.3）

**观察**:
- 源码中有 `showToast` 函数，用于错误提示。
- 没有看到实际错误提示样式。

**问题**:
- 无法验证错误提示的清晰度、可操作性、持续时间。
- 需要真实测试：串口占用、设备拔出、WSL 未安装等场景。

## 六、优先级建议

### 高优先级（立即处理）

1. **优化快捷指令入口**：将三横线图标改为"快捷指令 ▼"文字+图标组合，并预置示例指令。
2. **改善信息密度**：高级设置行默认折叠，仅保留核心参数；提供"常用配置"快速切换。
3. **提升 WSL 入口与引导**：为 WSL 功能设计更显眼的入口，并在首次打开时提供依赖检查清单。

### 中优先级（下版本）

4. **添加空状态提示**：主输出区首次启动时显示"连接串口后，数据将显示在这里"。
5. **调整主题风格入口**：考虑将风格选择移入设置菜单，标题栏仅保留深浅色切换。
6. **优化分栏布局**：分栏时自动切换为紧凑工具栏模式，避免拥挤。
7. **日志功能入口上浮**：将"保存日志"放到发送栏附近或工具栏常驻区域。

### 低优先级 / 需验证

8. **齿轮按钮状态同步**：根据高级设置行展开/折叠状态切换图标。
9. **辅助功能增强**：为图标按钮添加 aria-label、快捷键提示。
10. **onboarding 效果验证**：在真实用户测试中记录完成率与跳过步数。

## 七、本次走查的局限性与下一步

### 7.1 局限性

1. **无动态交互**：无法点击、输入、触发 onboarding，导致对动态反馈、错误提示、动画的评估依赖源码推断。
2. **单用户视角**：仅一个专家（本研究者）评估，未进行多专家独立评估（通常启发式评估建议 3-5 名专家）。
3. **单一环境**：仅在 Windows 11、1367×1035 窗口、默认主题下观察，未覆盖不同分辨率、DPI、高对比度模式。
4. **无真实任务数据**：未连接真实串口设备进行收发测试，对性能、卡顿、数据流显示效果的评估缺失。

### 7.2 下一步建议

1. **立即修复高优先级问题**：快捷指令入口、信息密度、WSL 入口，这些都是低成本高影响的改进。
2. **启动 P2 可用性测试**：用本次走查的修正版任务脚本（T1-T7）对真实用户进行测试，重点关注首屏可发现性、首次引导、WSL/工作流入口。
3. **P3 问卷补充**：在问卷中单独增加对"快捷指令发现率""工作流发现率""风格切换频率"的测量。
4. **配置 desktop-operator MCP**：后续如需深度界面交互测试，可在 WorkBuddy 中配置 `desktop-operator` MCP server，实现点击、输入、滚动等自动化操作。
