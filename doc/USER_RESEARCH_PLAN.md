# Seahi-Serial 用户研究计划与初步分析

> **研究主题**：嵌入式串口调试工具的用户需求与痛点洞察
> **研究对象**：Seahi-Serial v0.2.0（Tauri 2 + Rust 串口调试器）
> **研究周期**：8 周（分四阶段）
> **研究者**：UX Researcher
> **文档版本**：v1.1 | 2026-07-18（v1.1 更新：第七节替换为基于真实界面走查的修正版分析）

---

## 一、研究背景与动机

### 1.1 项目现状

Seahi-Serial 是一款面向嵌入式开发者的轻量级串口调试桌面工具，采用 Tauri 2 + Rust 技术栈，仅支持 Windows。当前版本 v0.2.0 已实现以下核心能力：

| 能力域 | 已交付功能 |
|--------|-----------|
| **基础串口** | 多串口分栏、完整配置（波特率/数据位/停止位/校验位）、DTR/RTS 控制 |
| **数据交互** | 文本/HEX 双模、ANSI 颜色解析、终端模式、发送历史、快速指令 |
| **辅助功能** | 自动重连、自动滚动、时间戳、行号、日志导出、消息回显 |
| **WSL 集成** | usbipd-win 端口映射、WSL 内串口监控、发行版管理 |
| **自动化** | v0.2.0 新增工作流规则（正则匹配 → 自动发送/切换信号/保存日志） |
| **体验层** | 12 种主题、9 步首次引导、自动更新、USB 插拔检测 |

### 1.2 研究动机

当前产品决策主要依赖开发者自身经验与代码评审反馈，存在三个核心风险：

1. **需求验证缺口** — 功能虽多，但未经验证哪些是"高频刚需"哪些是"低频锦上添花"。例如 12 种主题、6 种风格是否真正契合嵌入式工程师的审美偏好？自动化工作流的真实使用场景是什么？
2. **可用性盲区** — 已有代码评审发现 4 个 P0 级问题（Mutex 锁阻塞、读取无超时等），但这些是技术层面的。用户操作层面的痛点（如多监视器切换心智负担、WSL 配置门槛、HEX 模式输入体验）尚未被系统性识别。
3. **用户画像模糊** — "嵌入式开发者"是一个宽泛标签。STM32 调试者、ESP32 物联网开发者、BL602 模组测试员、CH32V203 电源控制工程师，他们的任务流、工具链、痛点差异显著。缺乏分层画像导致功能优先级判断缺乏依据。

### 1.3 业务影响假设

本研究预期在三个层面产生影响：
- **功能优先级**：基于用户任务频率与痛点强度，重新排序 v0.3.0+ 路线图
- **可用性提升**：识别并修复 Top 5 用户操作痛点，预期任务完成率提升 30%+
- **市场定位**：明确 Seahi-Serial 在串口工具生态（vs. SecureCRT、MobaXterm、SSCOM、PuTTY）中的差异化优势

## 二、研究目标与核心问题

### 2.1 主要研究问题（RQ）

| 编号 | 研究问题 | 关联决策 |
|------|---------|---------|
| **RQ1** | 嵌入式开发者在串口调试场景下的核心任务流是什么？哪些步骤最耗时、最易出错？ | 功能优先级、交互优化 |
| **RQ2** | Seahi-Serial 当前的核心功能中，哪些被高频使用、哪些被忽略？使用方式是否符合设计预期？ | 功能裁剪、引导优化 |
| **RQ3** | 用户在多串口监控、WSL 集成、自动化工作流等"高级功能"上的认知门槛与实际使用障碍是什么？ | 高级功能可发现性、文档策略 |
| **RQ4** | 相比 SecureCRT / SSCOM / PuTTY 等竞品，用户选择或弃用 Seahi-Serial 的关键因素是什么？ | 差异化定位、增长策略 |

### 2.2 次要研究问题

- **RQ5**：不同硬件平台开发者（STM32 / ESP32 / BL602 / CH32）的任务模式差异有多大？是否需要平台化定制？
- **RQ6**：首次引导（9 步聚光灯）是否有效降低了学习成本？用户在哪一步流失？
- **RQ7**：主题系统的 12 种风格对用户而言是"价值"还是"噪音"？工程师群体的审美偏好集中度如何？
- **RQ8**：自动化工作流（v0.2.0）的真实使用场景有哪些？规则配置是否过于复杂？

### 2.3 成功指标

| 维度 | 指标 | 目标值 |
|------|------|--------|
| **研究质量** | 研究发现被产品采纳率 | ≥ 80% |
| **可用性** | 核心任务完成率（连接→收发→日志） | ≥ 95% |
| **可用性** | 首次任务平均耗时 | 较基线降低 30% |
| **满意度** | SUS 系统可用性量表得分 | ≥ 75（良好） |
| **忠诚度** | NPS 净推荐值 | ≥ 40 |
| **功能认知** | 高级功能（WSL/工作流）自主发现率 | ≥ 50% |

## 三、研究方法体系（四阶段混合方法）

本研究采用**探索 → 验证 → 规模化 → 追踪**的四阶段混合方法设计，定性与定量结合，相互三角验证。

### 3.1 方法选择矩阵

| 阶段 | 方法类型 | 核心方法 | 样本量 | 回答的问题 | 周期 |
|------|---------|---------|--------|-----------|------|
| **P1 探索** | 定性 | 深度访谈 + 竞品分析 | 12-15 人 | RQ1、RQ4、RQ5 | 2 周 |
| **P2 验证** | 定性 | 可用性测试（思考发声） | 8-10 人 | RQ2、RQ3、RQ6 | 2 周 |
| **P3 规模化** | 定量 | 在线问卷 + 行为分析 | 150-300 份 | RQ2、RQ7、RQ8 | 2 周 |
| **P4 追踪** | 混合 | NPS 调研 + 使用埋点 | 全量用户 | 长期满意度 | 持续 |

### 3.2 阶段一：探索性研究（深度访谈 + 竞品分析）

**目标**：建立用户心智模型，理解真实任务流，识别隐性痛点。

**访谈结构**（60 分钟/人）：
1. **背景暖场**（5 分钟）：岗位、硬件平台、开发经验年限
2. **工具使用史**（10 分钟）：当前用什么工具？为什么选它？切换成本？
3. **任务回溯**（25 分钟）：最近一次串口调试的完整流程——从目标设定到结果确认，逐步还原。重点追问"在哪里卡住了""怎么解决的"
4. **痛点挖掘**（10 分钟）：对当前工具最不满意的 3 件事；曾经因为工具问题导致的"事故"
5. **功能期望**（10 分钟）：如果有魔法，你希望串口工具能做什么？

**竞品分析对象**：SecureCRT、MobaXterm、SSCOM、PuTTY、Tera Term、Arduino Serial Monitor
- 分析维度：功能覆盖、交互范式、定价、用户口碑（GitHub Issues / 论坛评价）

**样本论证**：12-15 人基于主题饱和原则（Guest et al., 2006 研究表明 12 个访谈可达 90%+ 主题饱和）。

### 3.3 阶段二：可用性测试（任务测试 + 思考发声）

**目标**：量化核心任务的可用性，定位操作层面的具体障碍。

**任务设计**（基于 TEST_CASES.md 真实场景）：

| 任务编号 | 场景描述 | 成功标准 | 观察重点 |
|---------|---------|---------|---------|
| **T1** | 首次启动 → 连接一个 USB 串口 → 接收数据 | 60 秒内看到数据流 | 引导有效性、端口选择路径 |
| **T2** | 切换 HEX 模式查看数据 → 发送一条 HEX 指令 | 正确发送并回显 | 模式切换认知、HEX 输入格式 |
| **T3** | 配置一条快速指令 "AT\r\n" → 一键发送 | 指令保存并成功发送 | 快捷指令可发现性 |
| **T4** | 开启第二监视器 → 同时监控两个串口 | 两个面板独立收发 | 多监视器心智模型 |
| **T5** | 打开 WSL 映射 → 映射一个 USB 设备到 WSL | WSL 内可见设备 | WSL 功能门槛 |
| **T6** | 配置一条自动化工作流规则 | 规则触发正确动作 | 工作流配置复杂度 |
| **T7** | 导出本次调试日志到指定目录 | 文件正确生成 | 日志功能可发现性 |

**采集指标**：
- 定量：任务完成率、首次完成时间、错误次数、求助次数
- 定性：思考发声记录、情绪标记（困惑/惊讶/满意/挫败）、SUS 量表

### 3.4 阶段三：规模化验证（在线问卷）

**目标**：将定性发现推广到更大样本，验证频率与分布假设。

**问卷结构**（预计 8-10 分钟）：
1. **人口统计**（3 题）：岗位、硬件平台、从业年限
2. **工具使用**（5 题）：当前主力工具、使用频率、切换意愿
3. **功能偏好**（8 题）：对 Seahi-Serial 各功能模块的重要性评分（5 级李克特量表）
4. **痛点确认**（5 题）：对 P1 访谈提炼的痛点进行频率与严重度评分
5. **竞品对比**（3 题）：选择/弃用 Seahi-Serial 的原因
6. **开放题**（2 题）：最希望增加的功能、最想吐槽的问题

**分发渠道**：安信可科技社区、嵌入式开发群组、GitHub README 引流、电子工程世界论坛

**样本论证**：150-300 份，基于 95% 置信度、±8% 误差范围，目标人群基数约数千级。

### 3.5 阶段四：长期追踪（NPS + 使用埋点）

**目标**：建立持续的用户反馈闭环。

- **NPS 调研**：版本更新后 7 天内弹窗调研（"你会向同事推荐 Seahi-Serial 吗？0-10 分"）
- **使用埋点**（需开发）：记录功能模块使用频率、会话时长、错误触发次数（匿名化，遵循隐私伦理）
- **反馈通道**：应用内"提交 Issue"按钮的转化率跟踪

## 四、参与者招募策略

### 4.1 目标用户画像（待验证假设）

基于项目背景与开发者经验，初步假设三类核心用户：

| 画像 | 占比假设 | 典型场景 | 关键需求 |
|------|---------|---------|---------|
| **A. 模组测试工程师** | 40% | AT 指令测试、量产烧录验证 | 快速指令、HEX 收发、日志留存 |
| **B. 嵌入式应用开发者** | 35% | STM32/ESP32 固件调试、日志分析 | 多串口并行、ANSI 颜色、终端模式 |
| **C. 物联网/WSL 开发者** | 25% | 跨平台调试、Linux 侧串口交互 | WSL 集成、自动化工作流 |

### 4.2 招募标准

**纳入标准**：
- Windows 10/11 用户（产品仅支持 Windows）
- 过去 3 个月内有串口调试经历
- 覆盖至少 3 种硬件平台（STM32、ESP32、BL602、CH32、GD32 等）
- 经验梯度：新手（<1 年）、中级（1-3 年）、资深（3 年+）各占约 1/3

**排除标准**：
- 仅使用 macOS/Linux（产品不支持）
- 无实际串口调试需求的学生（避免低质量反馈）

**多样性保障**：
- 性别：不设硬性配额，但记录并分析差异
- 行业：覆盖消费电子、工业控制、物联网、教育
- 地域：一线城市 + 二三线城市混合

### 4.3 招募渠道

| 渠道 | 预期产出 | 成本 |
|------|---------|------|
| 安信可科技内部团队 + 客户 | 5-6 人（深度访谈） | 低 |
| 嵌入式开发者微信群/QQ群 | 4-5 人 + 问卷分发 | 低 |
| GitHub Issues 历史反馈者 | 3-4 人（高价值） | 低 |
| 电子工程世界 / 21ic 论坛 | 问卷主要来源 | 中 |
| 用户访谈激励（可选） | 提升完成率 | 50-100 元/人 |

## 五、研究工具包

### 5.1 深度访谈大纲（P1 阶段）

**开场说明**（标准化）：
> "感谢您参与本次访谈。我们正在研究嵌入式开发者在串口调试中的真实体验。访谈约 60 分钟，全程录音仅用于研究分析，不会公开您的身份。没有对错答案，请按真实情况分享。"

**核心问题清单**：

**A. 背景与工具史**
1. 请简单介绍您的工作内容和主要开发的硬件平台。
2. 您做嵌入式开发多少年了？串口调试在您工作中的频率是？
3. 您目前主力使用哪个串口工具？用了多久？当初为什么选它？
4. 您是否同时安装了多个串口工具？什么场景下会切换？

**B. 任务回溯（关键）**
5. 请回忆**最近一次**串口调试的完整过程——您当时的目标是什么？从打开工具到完成任务，每一步做了什么？
6. 在这个过程中，哪个环节最顺利？哪个环节最卡？
7. 有没有因为工具问题导致调试失败或延误的情况？请详细描述。
8. 您平时会同时监控多个串口吗？什么场景需要？

**C. 痛点深挖**
9. 如果给当前主力工具打分（10 分制），您打几分？扣分项是？
10. 您最希望串口工具"别再做"的一件事是什么？最希望它"开始做"的一件事是什么？
11. 您是否使用过自动化功能（脚本、宏、规则）？为什么用或不用？

**D. Seahi-Serial 体验（如适用）**
12. 您是否使用过 Seahi-Serial？如果用过：第一印象是什么？哪个功能让您眼前一亮？哪个让您困惑？
13. 如果没用过：听到"VS Code 风格的串口工具 + WSL 集成 + 自动化工作流"，您的第一反应是什么？

**E. 结束**
14. 还有什么想补充的吗？
15. 如果后续有问卷或新版本测试，您愿意继续参与吗？

## 七、专家走查发现（基于真实界面与源码）

### 7.2 用户画像假设（走查修正版）

**画像 A：量产测试工程师（假设占比 40%）**

- **特征**：重复性测试为主，AT 指令验证、固件烧录后自检
- **高频任务**：发送固定指令序列 → 检查响应 → 记录日志
- **走查发现的新痛点**：
  - 快捷指令入口（三横线图标）可发现性极低，量产场景最依赖的功能反而最难找（F2/P-09）
  - 日志保存入口折叠在高级设置行，量产高频操作路径过长（P-04）
  - 默认 5 条空白快捷指令，无示例引导
- **Seahi-Serial 契合点**：快速指令、发送历史、自动化工作流（但入口均需优化）

**画像 B：固件开发者（假设占比 35%）**

- **特征**：调试为主，长时间监控日志，频繁收发调试数据
- **高频任务**：连接 → 长时间接收 → ANSI 日志阅读 → 偶尔发送命令
- **走查发现的新痛点**：
  - 主输出区空状态缺失，首次打开不知此处会显示数据（F5/P-08）
  - 高级设置行默认全部展开，对仅需默认配置的固件调试者是信息过载（F1）
  - 分栏后工具栏空间不足，多串口对比体验差（F8/P-13）
- **Seahi-Serial 契合点**：多监视器、ANSI 解析、终端模式

**画像 C：跨平台/WSL 开发者（假设占比 25%）**

- **特征**：在 Windows 开发但在 Linux 侧运行/调试
- **高频任务**：WSL 内串口交互、USB 设备映射
- **走查发现的新痛点**：
  - WSL 入口视觉权重低，与差异化核心功能地位不匹配（F4）
  - 依赖链长（usbipd + 管理员权限 + Python bridge）且入口不明显，双重流失风险
- **Seahi-Serial 契合点**：WSL 集成是该群体的核心差异化价值

### 7.3 用户旅程地图（走查修正版）

| 阶段 | 用户行为 | 触点 | 情绪 | 走查发现的痛点 |
|------|---------|------|------|---------|
| **1. 准备** | 插入开发板，打开工具 | 设备插拔检测 | 期待 | ✅ 端口友好名正确显示中文（优势，F7） |
| **2. 配置** | 选端口、设波特率 | 端口下拉、波特率输入 | 困惑 | ⚠️ 高级设置行默认全展开，信息过载（F1） |
| **3. 连接** | 点击开始监控 | 开始/停止按钮 | 专注 | ⚠️ 齿轮按钮状态与高级设置行不一致（F10） |
| **4. 接收** | 观察输出数据 | 输出区、查看模式 | 困惑 | ⚠️ 空状态缺失，不知此处会显示数据（F5） |
| **5. 分析** | 查找关键日志、切换 HEX | 时间戳、行号 | 焦虑 | ⚠️ 图标依赖 tooltip，含义不直观（F9） |
| **6. 发送** | 输入调试命令 | 发送栏、快速指令 | 挫败 | ⚠️ 快捷指令入口可发现性极低（F2） |
| **7. 多串口** | 开第二监视器对比 | 分栏、额外监视器 | 困惑 | ⚠️ 分栏后工具栏拥挤（F8） |
| **8. 归档** | 保存日志 | 日志导出 | 平静 | ⚠️ 保存日志入口折叠在高级设置行（P-04） |
| **9. 异常** | 设备断开/重连 | 自动重连、错误提示 | 挫败 | ⚠️ 无法动态验证错误提示质量 |

## 八、研究伦理与质量保障

### 8.1 伦理原则
- **知情同意**：所有参与者签署书面/电子同意书
- **自愿原则**：可随时退出，不影响任何关系
- **隐私保护**：数据去标识化，录音转录后销毁原始录音
- **利益冲突声明**：研究者与产品团队的关系透明披露

### 8.2 研究质量保障
- **三角验证**：每个核心发现至少由 2 种方法交叉确认
- **同行评审**：P1 主题分析由第二位研究员独立编码 20% 样本
- **成员检查**：关键发现回访 2-3 位参与者确认解读准确性
- **偏差控制**：访谈采用开放式问题，避免引导性提问；可用性测试不干预用户操作
